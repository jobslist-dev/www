3:T1a7a,<p>Most hiring managers who open your GitHub spend under two minutes there. They are not reading your code line by line, they are not counting your commits, and they are definitely not impressed by a wall of green squares. They&#39;re looking for a quick answer to one question: is there evidence this person builds real things and thinks about how they build them?</p>
<p>Almost every developer profile fails that test by accident — not because the work is bad, but because nothing on the page tells the story.</p>
<h2>What reviewers actually do in those two minutes</h2>
<p>In rough order:</p>
<ol>
<li>Read the profile README or bio, if there is one.</li>
<li>Scan pinned repositories for something that isn&#39;t a tutorial.</li>
<li>Open the most interesting one and read its README.</li>
<li>If the README is good, look at the code structure — file layout, tests, commit messages.</li>
<li>Follow a live link, if you gave them one.</li>
</ol>
<p>Notice how much of that is reading rather than code review. The written layer around your work is doing most of the persuading, and it&#39;s the layer most developers skip.</p>
<h2>Pin the right four repositories</h2>
<p>The default pinned selection — whatever you happened to star or fork — is where most profiles lose. Choose deliberately:</p>
<ul>
<li><strong>One thing you built and shipped</strong>, ideally with a live URL. A working product beats everything else on the page.</li>
<li><strong>One thing with real engineering depth</strong> — a non-trivial problem, a performance improvement, a tool you wrote because you needed it.</li>
<li><strong>One contribution to something you didn&#39;t start</strong>, if you have one. Evidence you can work inside someone else&#39;s conventions is genuinely rare and valued.</li>
<li><strong>One thing that shows range</strong> — a different language, a different layer, an experiment.</li>
</ul>
<p>Unpin the tutorial follow-alongs, the abandoned two-commit repos, and the bootcamp assignments that thousands of other candidates also have. A profile with four strong repositories is stronger than one with forty repositories of which four are strong, because reviewers don&#39;t dig.</p>
<h2>The README is the highest-leverage thing you own</h2>
<p>For a hiring reviewer, the README <em>is</em> the project. Most are a title and an install command, which wastes the entire opportunity.</p>
<p>A README that converts contains:</p>
<ul>
<li><strong>One sentence on what it does and who it&#39;s for</strong>, at the very top.</li>
<li><strong>A screenshot, GIF, or live link.</strong> Something they can see without cloning.</li>
<li><strong>Why you built it</strong> — the problem, in a sentence or two.</li>
<li><strong>The interesting technical decisions.</strong> This is the part reviewers actually care about: what you chose, what you rejected, and why.</li>
<li><strong>What you&#39;d do differently now.</strong> Nothing signals seniority faster than an honest limitations section.</li>
<li><strong>How to run it,</strong> working, tested on a clean machine.</li>
</ul>
<p>That &quot;decisions and trade-offs&quot; section is worth more than another thousand lines of code. It converts a repository from something they&#39;d have to evaluate into something you&#39;ve already explained.</p>
<h2>Signals reviewers read without mentioning</h2>
<ul>
<li><strong>Commit messages.</strong> &quot;fix&quot;, &quot;update&quot;, &quot;asdf&quot; repeated forty times reads as carelessness. Clear messages read as someone who has worked on a team.</li>
<li><strong>Whether tests exist at all.</strong> Nobody expects full coverage on a side project. Some tests separate you from most candidates instantly.</li>
<li><strong>Repository hygiene.</strong> A <code>.gitignore</code> that works, no committed secrets, no <code>node_modules</code>, a license.</li>
<li><strong>Commit history shape.</strong> Steady work over months reads as genuine; a single enormous initial commit tells them nothing about how you work.</li>
<li><strong>Whether issues and pull requests were used.</strong> Even solo, it shows familiarity with real workflow.</li>
</ul>
<h2>Things that don&#39;t matter as much as people think</h2>
<ul>
<li><strong>The contribution graph.</strong> Nobody serious hires on green squares, and streak-padding commits are transparent. Life happens; gaps are fine.</li>
<li><strong>Follower and star counts.</strong> Nice, irrelevant.</li>
<li><strong>Total repository count.</strong> Twelve unfinished projects is a weaker signal than two finished ones.</li>
<li><strong>Having contributed to a famous project.</strong> A typo fix in a big-name repo impresses far less than a substantial fix in an obscure one.</li>
</ul>
<h2>Use the profile README</h2>
<p>The profile README — the repository named after your username — renders at the top of your page, and most developers never create one. It costs ten minutes and frames everything below it.</p>
<p>Keep it to a short block: what you build, what you&#39;re currently working on, what you&#39;re looking for, and links to your best project, your site, and a way to contact you. That&#39;s all. Skip the animated banners, the badge collections, and the visitor counters, which reviewers uniformly ignore.</p>
<h2>If your best work is private</h2>
<p>Plenty of strong engineers have almost nothing public, because their best work belongs to employers. This is normal and reviewers know it. Options that work:</p>
<ul>
<li><strong>Write about it instead.</strong> A short post explaining a hard problem you solved — no proprietary detail — demonstrates the same thinking.</li>
<li><strong>Extract the generic part.</strong> The utility, the pattern, the tool, rebuilt cleanly and open-sourced.</li>
<li><strong>Build one deliberate public thing</strong> with the quality bar of your professional work.</li>
<li><strong>Say so in your application.</strong> &quot;My production work is private; here&#39;s a walkthrough of something comparable&quot; is a perfectly good answer.</li>
</ul>
<p>An empty GitHub is not disqualifying. An empty GitHub with no explanation, alongside a candidate who provided one, is a disadvantage you can remove cheaply.</p>
<h2>The takeaway</h2>
<p>Your GitHub is skimmed, not audited. Pin four deliberate repositories, write READMEs that explain your decisions rather than your install steps, keep commit hygiene tidy, and add a short profile README that tells reviewers what you&#39;re looking for. Two hours of curation makes the work you&#39;ve already done actually legible — which is the whole job of a profile.</p>
